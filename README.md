# Customer Churn Prediction

A full end-to-end machine learning project that predicts which telecom customers are likely to churn — and explains exactly why — using XGBoost, SHAP explainability, and a live Streamlit web app.

## Project Overview

Customer churn is one of the most expensive problems in the telecom industry. Acquiring a new customer costs 5–7x more than retaining an existing one. This project builds a machine learning system that:

- Predicts which customers are at risk of churning
- Quantifies the probability of churn for each customer
- Explains *why* each customer is flagged using SHAP values
- Translates predictions into actionable business recommendations

**Dataset**: IBM Telco Customer Churn (7,043 customers, 21 features)  
**Source**: [Kaggle — Telco Customer Churn](https://www.kaggle.com/datasets/blastchar/telco-customer-churn)

## Results

| Metric | Score |
|---|---|
| Recall |  75.7% |
| Precision | 48.8% |
| F1 Score | 59.3% |
| ROC-AUC | 0.814 |
| Decision threshold | 0.45 |

> Recall was prioritised as the primary metric. Missing a real churner (false negative) costs more than a false alarm — an undetected churner represents permanent revenue loss, while a false alarm results in an unnecessary but recoverable retention offer.

---

## Key Findings

**Top churn drivers identified by SHAP:**

1. **`charge_per_tenure`** — engineered feature: customers paying a lot relative to how long they have been with the company feel poor value. Strongest single predictor.
2. **`InternetService_Fiber optic`** — fibre customers churn at higher rates, likely due to higher costs and greater market competition.
3. **`num_services`** — engineered feature: customers with more add-on services are significantly less likely to churn. Every additional service acts as a retention anchor.
4. **`MonthlyCharges`** — price-sensitive customers are at higher risk.
5. **`Contract_Two year`** — the strongest protective factor. Two-year contract customers churn at only 2.8% vs 42.7% for month-to-month.

> Two of the top three most important features were engineered features that did not exist in the raw dataset, validating the feature engineering decisions and demonstrating that domain-informed feature creation can outperform raw data signals.

      │  RandomizedSearchCV (60 iterations, 5-fold CV)

## Technical Decisions

### Why XGBoost?
XGBoost handles the three hardest properties of churn data well — class imbalance via `scale_pos_weight`, non-linear relationships (churn risk spikes sharply in the first 3 months then levels off), and mixed feature types after one-hot encoding.

### Why threshold 0.45 instead of default 0.50?
Moving from 0.50 to 0.45 costs only 0.1 F1 points but gains 2.7 percentage points of recall — approximately 4 more churners caught per 100 customers. Since the cost of missing a churner (permanent revenue loss) outweighs the cost of a false alarm (unnecessary discount to a loyal customer), this tradeoff is justified.

### Why was the tuned model rejected?
The RandomizedSearchCV cross-validation F1 of 85.32% was measured on SMOTE-balanced training folds, not the real test set. The tuned model scored F1=59.8% and Recall=72.5% on the test set — worse than the default model. The tuned model had overfit to synthetic SMOTE patterns. The default model at threshold 0.45 was retained as it demonstrated better real-world generalisation.

### Why was `drop_first=True` used in one-hot encoding?
To prevent the dummy variable trap — perfect multicollinearity where one column is perfectly predictable from the others. Dropping the first category makes it the implicit reference baseline with no information loss.

### Why was the scaler fitted on training data only?
Fitting the scaler on all data before splitting would cause data leakage — the scaler's mean and std would be influenced by test data, making evaluation metrics artificially optimistic. The scaler was saved as a `.pkl` file so the Streamlit app applies the exact same transformation to new customer inputs.

## Business Recommendations

Based on SHAP analysis, the following interventions are recommended for at-risk customers:

| Driver | Recommended action |
|---|---|
| High charge per tenure | Offer early loyalty discount in months 1–3 |
| Month-to-month contract | Incentivise upgrade to annual contract |
| Fibre optic service | Review pricing or offer bundle discount |
| Few add-on services | Cross-sell security, backup or tech support |
| Electronic check payment | Incentivise switch to automatic payment |
| Short tenure | Trigger proactive onboarding programme |

## What I Learned

- **Class imbalance is a business problem, not just a technical one.** The choice of evaluation metric (recall over accuracy) reflects the real cost asymmetry between false negatives and false positives.
- **Feature engineering beat raw features.** Two engineered features (`charge_per_tenure`, `num_services`) ranked in the top 3 most important features, above all original dataset columns.
- **Cross-validation on SMOTE data can be misleading.** A tuned model achieving 85% CV F1 performed *worse* on the real test set than the default model. Always validate on untouched test data.
- **Thresholds are business decisions.** The default 0.50 threshold is arbitrary. Choosing 0.45 based on the recall/precision tradeoff is a deliberate, justified business decision.
- **Explainability closes the loop.** SHAP transforms predictions into actionable interventions — not just "this customer will churn" but "offer them a contract upgrade because that is their primary risk driver."
